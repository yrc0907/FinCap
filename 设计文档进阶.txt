《AI 自动解说剪辑系统最终统一方案（完整展开版，个人快速开发版，无省略、无歧义）》

第 0 章：这份方案的使用方式

这份方案不是思路草稿，不是概述，不是演示版。
这份方案的目的，是给 AI 开发使用。
所以整份方案必须做到以下几点：

1. 每一个模块都说清楚“为什么存在”
2. 每一个模块都说清楚“具体干什么”
3. 每一个模块都说清楚“输入是什么”
4. 每一个模块都说清楚“输出是什么”
5. 每一个模块都说清楚“推荐技术栈”
6. 每一个模块都说清楚“不能怎么做”
7. 每一个模块都说清楚“和前后模块怎么衔接”
8. 每一个模块都说清楚“失败后如何处理”
9. 每一个模块都说清楚“控制台日志要打印什么”
10. 不允许开发时根据模糊理解自己脑补模块行为

这份方案是“个人快速开发版”。
含义不是“做得很弱”，而是：
在不影响核心效果的前提下，砍掉很多没有必要的大而重的系统，只保留真正解决问题的核心骨架。

这份方案解决的是：
“从原始视频到最终解说成片”的完整闭环。

━━━━━━━━━━━━━━━━━━
第 1 章：系统到底要解决什么问题
━━━━━━━━━━━━━━━━━━

这套系统不是单纯做视频剪辑，也不是单纯写文案，也不是单纯做配音。
它要解决的是“自动解说视频”里最常见、最致命的几类错误。

第 1 类问题：切点错误
具体表现：
- 本来应该在某个真实变化点切开，系统却切晚了或切早了
- 当前段尾巴混入下一段内容
- 当前段前面没问题，最后几百毫秒开始脏掉

第 2 类问题：主体提前入画
具体表现：
- 当前段在讲 A
- 但画面尾部已经出现 B
- 文案和画面开始打架
这通常比普通镜头切错更恶心，因为用户很容易感知到“讲的不是眼前这个人”。

第 3 类问题：事件被切碎
具体表现：
- 一个完整动作被切成 3 段
- 一次完整搞笑因果被切成 4 段
- 一个对话和反应链条被切得支离破碎
结果就是文案像流水账，节奏很差。

第 4 类问题：文案、配音、画面时长不贴合
具体表现：
- 文案写得太长
- 配音还没讲完画面先切了
- 画面已经进入下一事件，配音还在讲上一事件
- TTS 自己带首尾静音和中间无意义停顿，浪费时长

第 5 类问题：整条视频虽然段段都能讲，但整体不顺
具体表现：
- 开头不抓人
- 爆点没前置
- 低价值片段说太多
- 高价值片段没留原片
- 角色称呼前后不一致
- 整片像流水账而不是一个完整视频

这套系统的目标，不是追求理论上 100% 零误差。
而是追求：
在个人可开发、可运行、可迭代的前提下，大幅减少这些错误。

━━━━━━━━━━━━━━━━━━
第 2 章：最终方案的大原则
━━━━━━━━━━━━━━━━━━

原则 1：不能先把视频真实切成固定时长片段再分析
禁止的错误理解：
- 先把视频切成 2 秒一段
- 再分别检测人物、场景、写文案

原因：
如果真实变化发生在 2 秒内部，先固定切开就已经把错误带进来了。
正确做法是：
先沿完整时间轴检测变化，再生成真实动态片段。

原则 2：先切对，再看对，再写对
切错的片段送去理解，理解一定容易被污染。
理解错了，再写文案，就会越走越偏。
所以顺序不能反：
必须先解决边界，再做理解，再做表达。

原则 3：以“事件段”为主要工作单位，不以固定秒数为单位
系统最终处理的不是“2 秒段”“3 秒段”。
系统最终处理的是“事件段”。
事件段可能长，可能短，但必须尽量和真实内容单位一致。

原则 4：局部视觉理解必须保留，全局重型视觉理解可以砍掉
个人快速版里，必须有局部视觉理解，因为每一段到底发生了什么必须看懂。
但不一定要做重型全局剧情树和重型视觉全局理解。
可以用“事件级九宫格 + 文本整合”去替代重型全局视觉理解。

原则 5：用户提示词只能影响表达，不影响事实
用户可以通过提示词影响：
- 文案风格
- 节奏
- 原片保留
- 爆点前置
但不能影响：
- 边界是否真实
- 画面里到底是谁
- 哪段到底发生了什么

原则 6：局部重跑优先，整片重跑默认禁止
如果第 8 段错了，优先修第 8 段。
只有在局部修复无法解决的极少数情况下，才考虑扩大范围。

原则 7：配音时长问题，优先靠“生成前预测”解决
不能把系统做成：
- 文案写完
- TTS 生成
- 不行
- 再重写
- 再配
- 再不行再来一遍
正确做法是：
文案生成后，先做时长预测，预测通过了再去 TTS。

原则 8：音频和视频的微小变速只做最后精修，不做主修复
变速可以加，但只能作为最后一层微调。
不能靠变速去拯救一个本来就切错、看错、写错的片段。

原则 9：所有关键步骤都必须打印中文控制台日志
日志不是可有可无，而是必要部分。
因为你要知道：
- 系统现在做到哪一层了
- 为什么这一段被重跑
- 为什么这一段被改边界
- 为什么这一段被判定失败

原则 10：这套方案优先追求“开发快、效果稳、可迭代”
不是优先追求“最重最炫的完整系统”。

━━━━━━━━━━━━━━━━━━
第 4 章：用户配置与模型配置层
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这一层必须单独存在。
原因不是“好看”，而是你明确要求：
用户必须能够自己控制：
- 文案风格
- 最终视频中原片保留比例
- 模型接入方式
- API 地址
- 是否走本地模型
- 是否走第三方 API
- 多个地址的测试与切换
- 批量处理参数
- 是否启用九宫格
- 参考图上传和命名

如果没有这一层，这些能力就会散落在后端逻辑里，后续维护会非常乱。

二、这一层具体干什么

这一层负责读取、管理和校验所有用户级配置。

它必须支持的配置种类：

1. 内容类型
例如：
- 对话番
- 动作番
- 无对白动画
- 电影/电视剧
- 通用模式

2. 文案风格提示词
例如：
- 偏抖音风
- 偏 TikTok 风
- 偏剧情旁白
- 偏搞笑
- 偏高燃
- 用户自定义长提示词

3. 编排提示词
例如：
- 爆点优先放开头
- 高价值片段多留原片
- 过渡段尽量压缩
- 开头先放原片再进解说

4. 原片保留比例
例如：
- 整体保留 20%
- 整体保留 35%
- 高价值段额外多保留
- 爆点段强制先放原片

5. 模型与服务接入配置
包括：
- 视觉理解 API 地址
- 文案生成 API 地址
- TTS API 地址
- 多个备用地址
- 本地模型路径或本地服务地址
- 模型名
- 鉴权信息
- 是否启用

6. 批量处理配置
包括：
- 视觉理解批量大小
- 文案生成批量大小
- 并发数量
- 超时
- 失败重试次数

7. 校验与重跑配置
包括：
- 校验严格度
- 最大重跑次数
- 是否启用局部重跑
- 是否启用九宫格复杂段理解

三、这一层不能做什么

这一层不能决定视频事实。
这句话必须写死：

用户提示词和用户偏好，不能决定以下事情：
- 真实边界在哪里
- 当前片段的主主体是谁
- 画面中是否真的发生了某个动作
- 某一段是否真的属于某个事件

这些都必须由后面的检测和理解模块决定。

四、推荐技术栈

- 前端配置页面
- 后端配置存储
- API 健康检查接口
- 本地模型适配器
- 多地址优先级切换逻辑

五、控制台中文日志要求

必须打印：
- 【开始】读取用户配置
- 【过程】内容类型、风格、原片比例、模型来源
- 【过程】API 测试结果
- 【完成】用户配置加载完成
- 【错误】用户配置校验失败
- 【回退】加载默认配置或最近一次成功配置

━━━━━━━━━━━━━━━━━━
第 5 章：参考图片与命名增强层（可选但建议保留）
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

你明确提出：
用户可以上传图片并编辑名字，以提高视觉理解的准确性。

这个能力对以下内容非常有帮助：
- 动画角色
- 真人多角色
- 相似角色
- 物体型主体
- 道具型主体
- 场景型主体

例如：
如果用户上传了“哆啦A梦”“大雄”“大雄爸爸”的参考图并命名，
后面的视觉理解和文案命名就会稳定很多。

二、这一层具体干什么

1. 接收用户上传的参考图片
2. 保存图片
3. 保存用户给图片设置的名字
4. 保存图片类别
   - 人物
   - 动物
   - 物体
   - 场景
   - 道具
5. 在后续视觉理解时，把这些参考信息作为辅助上下文传给模型

三、这一层的真实定位

这一层是“增强层”，不是“裁判层”。

意思是：
- 它可以帮助模型更稳地识别角色
- 它可以帮助文案统一叫法
- 它可以帮助减少主体识别漂移

但它不能变成绝对真值。
如果参考图和实际画面冲突，系统不能盲信参考图。

四、推荐技术栈

- 图片上传
- 图片缓存
- 参考图管理
- 将参考图作为视觉模型多图输入的一部分或上下文说明

五、控制台中文日志要求

- 【开始】加载参考图片与命名
- 【过程】参考图片数量、命名数量、图片类别统计
- 【完成】参考增强层加载完成
- 【警告】存在命名冲突或重复
- 【错误】参考图加载失败
- 【回退】跳过参考增强，继续主流程

━━━━━━━━━━━━━━━━━━
第 6 章：轻量时间轴与缓存管理
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这不是重型“全视频元数据索引库”，但也不能没有任何时间和缓存管理。

你前面提过一个问题：
“如果我要重跑某一个片段，我直接跑这个时间段不就行了，还需要索引层吗？”
答案是：
你不需要重型索引层，但仍然需要轻量时间轴和缓存管理。

原因如下：
1. 你要知道某个时间段对应的是视频的哪一部分
2. 你要知道某个片段之前有没有已经抽好的关键帧
3. 你要知道某个片段是否已有检测缓存
4. 你要知道某个片段重跑后要覆盖哪些旧结果

所以这一层必须存在，但只做轻量版本。

二、这一层具体干什么

1. 读取视频总时长
2. 读取帧率
3. 读取分辨率
4. 建立大致的时间到视频资源的映射
5. 给局部片段建立缓存目录
6. 缓存以下局部结果：
   - 抽帧结果
   - 边界检测结果
   - 局部视觉理解结果
   - 文案结果
   - TTS 结果

三、这一层不做什么

1. 不做全视频每一帧的重型统一特征库
2. 不做全量帧级深特征索引
3. 不做复杂数据库式管理
4. 不做脱离当前项目需要的大基础设施

四、推荐技术栈

- ffprobe
- FFmpeg
- OpenCV
- 本地文件缓存

五、控制台中文日志要求

- 【开始】建立轻量时间轴与缓存管理
- 【过程】视频时长、帧率、分辨率
- 【过程】缓存目录、已有缓存数量
- 【完成】轻量时间轴与缓存管理完成
- 【错误】缓存目录不可用
- 【回退】使用临时缓存目录

━━━━━━━━━━━━━━━━━━
第 7 章：基于完整时间轴的双信号边界检测
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这是整套系统里最核心的一层之一。
它解决的是：
“真实边界到底在哪里？”

这里必须再次明确：
这一步不能先真实切成固定 2 秒片段再分别检测。
正确做法是：
沿完整时间轴扫描，寻找变化，再决定边界。

二、为什么叫“双信号”

因为只看一种信号不够稳。
这一层必须同时看两种变化：

1. 主体变化
2. 镜头 / 场景变化

三、主体变化检测到底看什么

它重点看：
1. 当前画面主体是谁
2. 新主体是否稳定进入核心区域
3. 原主体是否退出核心区域
4. 主体集合是否发生变化
5. 当前段尾巴是否已经开始被下一段主体污染

这一层尤其适合解决：
“当前段在讲 A，但尾部已经出现 B”的问题。

四、镜头 / 场景变化检测到底看什么

它重点看：
1. 镜头有没有切换
2. 场景有没有变化
3. 画面整体统计分布有没有突变
4. 运动模式有没有变化
5. 当前叙事是否已经进入新阶段

这一层尤其适合解决：
- 明显镜头切换
- 场景切换
- 动作节奏变化
- 画面整体结构变化

五、为什么两种信号都要保留

只看主体变化，会漏掉：
- 主体没换，但场景切了
- 主体没换，但叙事阶段已经切换

只看镜头变化，会漏掉：
- 场景没明显切，但下一段主体已经进来了

所以两种必须一起保留。

六、这层具体怎么工作

1. 沿完整时间轴扫描
2. 主体变化检测给出候选变化点
3. 镜头 / 场景变化检测给出候选变化点
4. 两类候选点融合
5. 得到真实边界候选

七、推荐技术栈

主体变化检测：
- YOLO11
- ByteTrack 或 BoT-SORT
- 可选参考图增强

镜头 / 场景变化检测：
- PySceneDetect
- OpenCV 直方图比较
- OpenCV 光流
- OpenCV 帧差

八、重要规则

1. 主体变化优先解决“提前泄露”
2. 镜头 / 场景变化负责兜底
3. 单一检测器不能直接下最终边界结论
4. 低置信结果不能强切

九、控制台中文日志要求

- 【开始】边界检测开始
- 【过程】主体变化候选点数量
- 【过程】镜头 / 场景变化候选点数量
- 【过程】融合后的边界候选数量
- 【完成】边界检测完成
- 【警告】存在低置信边界
- 【错误】边界检测失败
- 【回退】保留当前边界继续后续流程

━━━━━━━━━━━━━━━━━━
第 8 章：边界融合、边界修正与动态片段生成
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

边界检测层只能告诉你：
“这里附近可能该切。”
但真正把这些候选点变成“可以实际使用的片段”，是这一层的工作。

二、这一层具体干什么

1. 合并时间很接近的边界点
2. 过滤明显无效点
3. 根据边界生成动态片段
4. 保证片段首尾连续、无重叠、无空缺
5. 优先修片段尾部，不随便大动前半段
6. 避免片段变得极端过短或极端过长

三、为什么这里不单独保留“核心区 / 容错区修正层”作为重模块

因为个人快速版没必要搞成重模块。
但“优先修尾部”“前半段尽量不动”的思想仍然保留，作为修正规则存在。

正确写法是：
- 这是边界修正规则
- 不是一个独立超重模块

四、这一层的输出是什么

输出的是：
真正可用的动态片段列表。

也就是说，最终不再是固定 2 秒一段，
而是根据真实边界形成的：
- 0.00 - 1.42
- 1.42 - 2.68
- 2.68 - 3.50
- 3.50 - 6.00
这种非均匀片段。

五、推荐技术栈

- 时间轴规则逻辑
- 合法性检查器
- 简单平滑策略

六、控制台中文日志要求

- 【开始】动态片段生成开始
- 【过程】候选边界数量
- 【过程】最终动态片段数量
- 【过程】最短片段、最长片段
- 【完成】动态片段生成完成
- 【警告】存在异常短片段
- 【错误】片段生成失败
- 【回退】退回最近可用边界版本

━━━━━━━━━━━━━━━━━━
第 9 章：事件合并
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

动态片段虽然比固定切片好多了，但仍然可能偏镜头级。
也就是说：
一个完整事件，还是可能被切成多个小片段。

所以必须做事件合并。

二、这一层具体干什么

它要把属于同一个语义事件的多个动态片段合并成一个更完整的事件段。

例如：
- 一个动作过程
- 一个对话 + 反应过程
- 一个搞笑因果链
- 一个完整的情绪推进段

三、为什么必须保留

如果不做事件合并，会出现：
- 文案太碎
- 节奏太乱
- 像逐镜头流水账

四、这一层根据什么合并

合并依据包括：
1. 主体是否连续
2. 场景是否连续
3. 运动是否连续
4. 内容类型规则

不同内容类型规则不同：

对话番：
- 减少反打碎裂

动作番：
- 合并完整动作链

无对白动画：
- 合并动作因果链

电影 / 电视剧：
- 保留情绪推进的完整性

五、推荐技术栈

- 规则引擎
- 主体变化信息
- 场景变化信息
- 运动变化信息
- 内容类型规则

六、控制台中文日志要求

- 【开始】事件合并开始
- 【过程】合并前动态片段数量
- 【过程】合并后事件段数量
- 【完成】事件合并完成
- 【警告】存在可疑碎片化事件段
- 【错误】事件合并失败
- 【回退】保留动态片段直接继续

━━━━━━━━━━━━━━━━━━
第 10 章：关键帧 / 关键片段选择
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

你不可能把一个事件段的所有帧都送给视觉模型。
那样太重、太慢、太贵。

但也不能只给一张图，因为一张图通常不够看懂整个事件。

所以必须有一层：
专门为视觉理解准备“最有信息量的输入”。

二、这一层具体干什么

从每个事件段中选择：
- 起点代表帧
- 峰值代表帧
- 结果代表帧
- 必要时的反应帧

对于高复杂度事件段，可以升级成：
- 少量关键短片段

三、为什么必须保留

这层不是视觉理解本身，
它是视觉理解的输入准备层。

如果没有它：
- 输入太少，看不懂
- 输入太多，成本过高
- 重复画面太多，浪费 token

四、推荐技术栈

- OpenCV
- FFmpeg 抽帧
- 帧差辅助
- 光流辅助
- 去重逻辑

五、控制台中文日志要求

- 【开始】关键帧 / 关键片段选择开始
- 【过程】当前事件段编号、抽取数量
- 【过程】高复杂度事件段数量
- 【完成】关键帧 / 关键片段选择完成
- 【警告】关键帧信息不足
- 【错误】抽帧失败
- 【回退】退回低成本抽帧模式

━━━━━━━━━━━━━━━━━━
第 11 章：事件级九宫格构建（可选，对复杂段启用）
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

你前面明确提出了一个很好的思路：
如果有支持多图输入的大模型，能不能把一段完整剧情做成九宫格，让模型一次看懂这一段。

答案是：可以，而且很适合复杂事件段。

但必须讲清楚：
九宫格不是替代一切的新系统。
它只是视觉输入方式的升级版。

二、这一层具体干什么

对于复杂事件段，
把这一段的代表画面拼成一张九宫格图，或者类似的多图集合。
然后把这张九宫格图送给支持多图输入的视觉模型，让模型一次理解“这一整段”的大意。

三、它适合什么情况

适合：
- 一个事件段内部有起承转合
- 一个事件段内部有主体变化
- 一个事件段内部有明显过程和结果
- 动作段
- 搞笑因果段
- 剧情推进段

四、它不适合替代什么

它不能替代：
- 边界检测
- 事件合并
- 关键帧选择
- 结构化摘要
- 文本层的全局整合

它只能替代或补强：
- 复杂事件段的视觉理解输入方式

五、推荐技术栈

- 关键帧拼图逻辑
- 支持多图输入的视觉模型
- 可选：九宫格或多图集合输入

六、控制台中文日志要求

- 【开始】九宫格构建开始
- 【过程】当前事件段编号、拼图图片数量
- 【完成】九宫格构建完成
- 【警告】复杂段信息不足，九宫格效果可能不稳定
- 【错误】九宫格构建失败
- 【回退】退回普通关键帧理解模式

━━━━━━━━━━━━━━━━━━
第 12 章：局部视觉理解 / 九宫格事件理解
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这是当前版本必须保留的核心层。
因为你无论如何都得知道：
当前事件段里到底发生了什么。

二、这一层具体干什么

它负责回答：
- 当前段主体是谁
- 当前段做了什么
- 当前段谁在反应
- 当前段场景是什么
- 当前段情绪是什么
- 当前段结果是什么

三、这一层分两种模式

模式 A：普通局部视觉理解
适合简单段。
输入：
- 若干关键帧
- 或少量关键短片段

模式 B：九宫格事件理解
适合复杂段。
输入：
- 事件级九宫格
- 或多图集合

四、为什么不做重型全局视觉理解

因为你现在是个人快速开发版。
重型全局视觉理解会把开发难度和成本抬太高。
这一版先保证“每个事件段尽量看对”，就已经能解决大部分核心问题。

五、推荐技术栈

- VLM API
- 关键帧输入
- 多图输入
- 九宫格输入
- 参考图增强（可选）

六、控制台中文日志要求

- 【开始】局部视觉理解开始
- 【过程】当前事件段编号、输入类型（关键帧 / 九宫格）
- 【过程】主体、动作、情绪、结果、置信度
- 【完成】局部视觉理解完成
- 【警告】当前段理解置信度低
- 【错误】局部视觉理解失败
- 【回退】切换低成本理解模式或请求重试

━━━━━━━━━━━━━━━━━━
第 13 章：结构化摘要
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

视觉理解和文案生成必须分开。
视觉理解负责“看懂”，文案生成负责“说出来”。

这一层就是把“看懂”的结果标准化。

二、这一层具体干什么

把局部视觉理解结果转成标准化中间结果。
至少要包含：
- 当前段主体
- 当前段主动作
- 当前段场景
- 当前段情绪
- 当前段结果
- 当前段置信度
- 当前段不确定说明

三、为什么必须保留

如果没有这一层：
- 文案生成会混乱
- 一致性校验会很难做
- 局部重跑会很难精确
- 用户改风格会污染事实层

四、这一层不做什么

它不直接写文案。
它也不做最终排序。
它只负责标准化事实表示。

五、控制台中文日志要求

- 【开始】结构化摘要生成开始
- 【过程】摘要关键字段
- 【过程】置信度
- 【完成】结构化摘要生成完成
- 【错误】结构化摘要生成失败
- 【回退】保留原始理解结果进入人工降级模式

━━━━━━━━━━━━━━━━━━
第 14 章：轻量全局剧情整合（文本层，可选但建议保留）
━━━━━━━━━━━━━━━━━━

一、这一层为什么存在

你前面质疑过一个非常关键的问题：
“如果视觉模型本身只能看单帧或少量图片，它怎么把整条视频串起来？”

答案是：
它本身做不到。
真正把整条视频串起来的，不是视觉模型，而是文本模型。

所以这一层不是“全局视觉理解层”，而是：
“轻量全局剧情整合层”。

二、这一层具体干什么

输入：
按时间顺序排列的所有事件段结构化摘要。

输出：
- 整条视频的一句话或几句话主线总结
- 主要角色列表
- 哪些段是高价值段
- 各段大致属于：铺垫 / 推进 / 爆点 / 结果

三、为什么建议保留

虽然它不是绝对必须，
但它实现成本不算高，收益却很明显：
- 帮助统一角色称呼
- 帮助标记高价值段
- 帮助文案层更连贯
- 帮助原片比例控制更合理

四、为什么不把它做成重型全局剧情树

因为个人快速版不需要重型树系统。
轻量整合已经够用了。

五、推荐技术栈

- 文本大模型
- 结构化摘要拼接
- 简单规则标签

六、控制台中文日志要求

- 【开始】轻量全局剧情整合开始
- 【过程】事件段数量、主角色数量
- 【过程】主线摘要、高价值段数量
- 【完成】轻量全局剧情整合完成
- 【错误】全局剧情整合失败
- 【回退】跳过全局整合，直接进入文案层

━━━━━━━━━━━━━━━━━━
第 15 章：文案生成
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这一层负责真正写解说文案。
前面的层都在做“看懂和整理”，这一层开始做“表达”。

二、这一层具体干什么

输入包括：
- 当前段结构化摘要
- 用户提示词
- 内容类型
- 文案风格
- 当前片段时长
- 可选的轻量全局剧情整合结果
- 当前段是否高价值
- 当前段是否建议先保留原片

输出包括：
- 当前段文案
- 当前段建议语速
- 是否建议先放原片再进解说

三、这一层必须遵守的规则

1. 文案必须服从片段时长
2. 文案风格受用户提示词影响
3. 高价值段允许少讲、先放原片
4. 低价值段可以更短
5. 文案不能脱离当前段事实
6. 文案不能越界讲到后面还没出现的画面

四、推荐技术栈

- 文案大模型
- 风格模板
- 提示词注入
- 可选全局摘要辅助

五、控制台中文日志要求

- 【开始】文案生成开始
- 【过程】当前段编号、风格、文案长度
- 【过程】是否高价值段、是否原片优先
- 【完成】文案生成完成
- 【错误】文案生成失败
- 【回退】重试或退回简化文案模板

━━━━━━━━━━━━━━━━━━
第 16 章：文案时长预测
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这一层非常关键。
你已经明确提出：
不希望出现“先生成 TTS，再发现超时，再回去改文案，再生成，再失败”的循环。
所以必须在 TTS 之前先做时长预测。

二、这一层具体干什么

输入：
- 当前段文案文本
- 目标片段时长
- 音色
- 语速档位
- 风格

输出：
- 预测朗读时长
- 与目标时长的差值
- 是否通过
- 是否需要重写文案

三、这一层不是做什么

它不是去计算“每一个字精确要几秒”。
因为一个字的发音时长会被上下文、停顿、语气、音色影响，不能按单字精确查表。
它做的是：
对整句 / 整段文案做时长预测。

四、推荐做法

1. 先用规则做基础估时
2. 再基于你自己的常用 TTS 音色做离线标定
3. 得到一个适合你自己系统的时长预测器

五、为什么这层必须保留

因为它能极大减少：
- 文案反复改
- TTS 反复生成
- 生成后才发现超时

六、控制台中文日志要求

- 【开始】文案时长预测开始
- 【过程】目标时长、预测时长、差值
- 【完成】文案时长预测完成
- 【警告】预测超时，退回文案重写
- 【错误】时长预测失败
- 【回退】使用保守文本长度阈值继续

━━━━━━━━━━━━━━━━━━
第 17 章：一致性校验
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这一层是质检层。
它负责判断：
“当前这一段，现在这样到底能不能直接进配音和合成？”

二、这一层具体干什么

至少检查三件事：

1. 主体一致性
当前画面里主要是谁，文案是不是在讲同样的主体。

2. 语义一致性
文案说的事，和画面表达的是不是同一件事。

3. 时长一致性
文案预测时长和片段时长是不是大致匹配。

三、为什么必须保留

因为前面不可能 100% 零误差。
没有这层，错误会直接流到成片。

四、输出必须包括

- 是否通过
- 失败原因码
- 失败原因说明

五、推荐技术栈

- 规则校验
- 轻量图文一致性打分
- 必要时廉价 VLM 复核

六、控制台中文日志要求

- 【开始】一致性校验开始
- 【过程】主体校验结果、语义校验结果、时长校验结果
- 【完成】一致性校验完成
- 【错误】一致性校验失败，原因码：……
- 【回退】交给局部重跑层

━━━━━━━━━━━━━━━━━━
第 18 章：自修复 / 局部重跑
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这一层是返修层。
如果一致性校验不过，系统不能停，也不能默认整片重来。
正确做法是只修当前问题段。

二、这一层具体干什么

固定修复顺序建议如下：

第一步：重写文案
如果问题主要是：
- 文案太长
- 文案讲偏了
- 文案过密

优先重写文案。

第二步：重跑当前段局部视觉理解
如果怀疑是当前段理解有偏差，就重新看这一段。

第三步：重切当前段及相邻段
如果怀疑问题来自边界污染，就回到边界层重切当前段与相邻段。

第四步：必要时重做事件合并
如果某段太碎或事件边界错了，就重做当前段附近的事件合并。

三、这一层必须限制的东西

- 最大重跑次数
- 不允许死循环
- 达上限必须进入保守输出或失败记录

四、推荐技术栈

- 重试调度逻辑
- 缓存复用
- 错误类型系统

五、控制台中文日志要求

- 【开始】自修复 / 局部重跑开始
- 【过程】当前片段编号、当前重跑次数、重跑类型
- 【完成】自修复 / 局部重跑完成
- 【错误】局部重跑失败
- 【回退】进入保守输出或跳过该段

━━━━━━━━━━━━━━━━━━
第 19 章：TTS
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这一层负责把最终通过校验的文案变成音频。

二、这一层必须遵守的原则

1. TTS 不是主修复手段
2. 大误差不应该靠 TTS 去救
3. TTS 只负责把合格文案转成音频，并做小幅可控的语速变化

三、推荐技术栈

- 你的 TTS 服务
- 音色配置
- 语速档位控制

四、控制台中文日志要求

- 【开始】TTS 生成开始
- 【过程】当前片段编号、音色、语速档位
- 【完成】TTS 生成完成
- 【错误】TTS 生成失败
- 【回退】切换备用音色或备用 TTS 服务

第 20 章：TTS 静音与气口清理
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

你明确提出：
TTS 配完以后会有首尾静音，中间还有无意义气口，这些都很浪费时长。

这层就是专门清理这些东西的。

二、这一层具体干什么

1. 清理首尾静音
这是必须做的。
因为首尾静音对内容没有价值，只会浪费时长。

2. 清理中间无意义停顿
注意，不是删除所有中间停顿。
只删除：
- 过长无意义气口
- 明显异常空白
- 不属于语义断句的多余停顿

3. 保留有意义的停顿
必须保留：
- 合理断句
- 情绪停顿
- 转折停顿
- 强调停顿

三、为什么必须有这一层

这层能明显减少轻微时长不匹配。
而且它比重写文案成本低。

四、推荐技术栈

- 音频静音检测
- 音频裁剪处理
- 中间停顿规则清理

五、控制台中文日志要求

- 【开始】TTS 静音与气口清理开始
- 【过程】首尾静音时长、中间清理次数
- 【完成】TTS 静音与气口清理完成
- 【警告】发现过多异常停顿
- 【错误】音频清理失败
- 【回退】保留原始 TTS 音频继续

━━━━━━━━━━━━━━━━━━
第 21 章：音频 / 视频微小变速对齐
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

有些片段已经基本正确了，但还是差一点点。
例如只差 0.05 秒、0.08 秒、0.1 秒。
为了这种小误差去重写文案或重切边界太浪费。

所以加这一层做最后精修。

二、这一层具体干什么

在片段已经大致正确的前提下，
对音频和视频做微小动态变速，使最终贴合更好。

三、必须强调的规则

1. 只能做微小变速
2. 音频和视频都可以承担一点点调整
3. 不能大幅拉伸
4. 不能拿它去救语义错误
5. 不能拿它去救严重时长错误

四、推荐策略

- 音频只允许很小范围调整
- 视频只允许更小范围调整
- 复杂动作段、口型敏感段更保守
- 高价值段更保守

五、推荐技术栈

- 音频变速处理
- 视频变速处理
- 时长差值控制逻辑

六、控制台中文日志要求

- 【开始】音频 / 视频微小变速对齐开始
- 【过程】音频倍率、视频倍率、对齐前后差值
- 【完成】音频 / 视频微小变速对齐完成
- 【警告】差值过大，不适合进入微调层
- 【回退】退回上游修复层

━━━━━━━━━━━━━━━━━━
第 22 章：原片比例控制
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

你明确要求：
用户必须能够自己调整最终成片中原片播放比例。

二、这一层具体干什么

它控制：
- 整条视频中原片直接播放的比例
- 高价值段是否多保留原片
- 爆点段是否先放原片再进解说
- 普通段是否缩短原片

三、这一层不做什么

它不参与：
- 边界检测
- 事件判断
- 视觉理解事实判断

它只参与：
- 合成前的时间分配
- 原片 / 解说比例规划

四、推荐做法

个人快速版只做简化版：
1. 全局原片比例参数
2. 高价值段额外加权
3. 爆点段优先保留原片开关

五、控制台中文日志要求

- 【开始】原片比例控制开始
- 【过程】全局原片比例、高价值段额外比例
- 【完成】原片比例控制完成
- 【警告】原片比例过高可能压缩解说空间
- 【回退】退回默认比例

第 23 章：合成
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这一层负责把所有内容真正变成成片。
前面都只是准备和修正。
这里才是真正输出。

二、这一层具体干什么

输入：
- 最终片段时间轴
- 配音音频
- 字幕
- 原片保留策略
- 微调后音视频时长

输出：
- 最终成片

三、合成原则

1. 默认普通硬切
2. 不默认复杂转场
3. 原片优先段允许先放原片再接解说
4. 至少做到段级严格对齐

四、推荐技术栈

- FFmpeg

五、控制台中文日志要求

- 【开始】视频合成开始
- 【过程】片段数量、字幕状态、音轨状态
- 【完成】视频合成完成
- 【错误】视频合成失败
- 【回退】导出调试版中间结果

第 24 章：最终验收
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

这是最后一道总检查。
前面每层可能局部正确，但这里要确保整片结构上可交付。

二、这一层具体干什么

检查内容包括：
1. 是否存在重叠片段
2. 是否存在空缺片段
3. 是否存在明显无效极短段
4. 是否能正常导出
5. 是否有明显时长风险残留

三、这一层为什么不能删

因为合成成功不等于结构一定正确。
必须有总验收。

四、推荐技术栈

- 时间轴规则检查
- 导出校验
- 最终结构检查

五、控制台中文日志要求

- 【开始】最终验收开始
- 【过程】重叠数量、空缺数量、无效极短段数量
- 【完成】最终验收完成
- 【错误】最终验收未通过
- 【回退】返回出错片段列表供局部重跑

第 25 章：批量处理怎么兼容进来
━━━━━━━━━━━━━━━━━━

一、这一层存在的原因

你明确要求支持批量用 API 跑：
- 多段视觉理解
- 多段文案生成
- 多段 TTS

二、个人快速版怎么做

不做复杂批量调度平台。
只做轻量批量处理能力：

1. 多个事件段的局部视觉理解可批量提交
2. 多个事件段的文案生成可批量提交
3. 多个事件段的 TTS 可批量提交
4. 单段失败不拖垮整批
5. 失败段单独重跑

三、这一层只改变什么

它只改变：
- 调用方式
- 并发方式

它不改变：
- 单段逻辑
- 校验规则
- 重跑规则

四、控制台中文日志要求

- 【开始】批量任务调度开始
- 【过程】批次大小、并发数量、成功数量、失败数量
- 【完成】批量任务调度完成
- 【错误】某批次失败
- 【回退】将失败任务放入重试队列

第 26 章：哪些模块被砍掉了，为什么砍
━━━━━━━━━━━━━━━━━━

为了让这版适合个人快速开发，以下内容被砍掉或降级：

1. 重型视频基础索引层
原因：
太重，个人版不需要全视频深特征索引。
替代：
轻量时间轴与缓存管理。

2. 完整时间轴多尺度检测层（作为独立大模块）
原因：
会让结构看起来重复。
替代：
只保留“沿完整时间轴检测”的原则，融进边界检测层。

3. 重型全局视觉理解层
原因：
视觉模型本身不能直接看完整片并构建完整剧情树，做重了不划算。
替代：
事件级九宫格理解 + 文本层轻量全局剧情整合。

4. 全局事件树大系统
原因：
过重。
替代：
轻量全局摘要。

5. 角色状态流大系统
原因：
过重。
替代：
结构化摘要 + 可选全局角色列表。

6. 重型导演层
原因：
个人快速版优先解决“能对上”，不是优先做“最像专业剪辑师”。
替代：
仅保留原片比例控制和提示词对文案/排序的轻量影响。

7. 自动自学习系统
原因：
开发重，收益滞后。
替代：
日志和缓存即可。

第 28 章：最后一句总定义
━━━━━━━━━━━━━━━━━━

这套最终统一方案的核心，不是做一个最重最全的大而全系统，
而是保留真正有效的骨架：

先沿完整时间轴检测真实边界，
再生成动态事件段，
再用关键帧或九宫格看懂当前事件，
再把理解结果标准化，
再生成受用户提示词控制但服从事实的文案，
再在配音前做时长预测，
再通过校验、局部重跑、TTS 清理和微小变速，把最终音画贴合度尽量做高，
最后再按用户要求控制原片比例并输出成片。

这就是最适合你现在个人快速开发、又能真正解决大部分音画不匹配问题的最终统一方案。
