《AI 自动解说剪辑最终方案（从零开始版，清晰无歧义）》

一、方案目标

开发一套自动解说剪辑系统。
系统输入一条原始视频，输出一条已经完成：
- 分段
- 理解
- 文案
- 配音
- 合成
- 校验
的最终成片。

这套系统必须解决的核心问题有四个：

1. 画面切分点不准确，导致一段里混入下一段内容。
2. 一个完整事件被切碎，或者多个事件被错误混在同一段里。
3. 文案、画面、配音三者表达的不是同一件事。
4. 整条视频虽然每段都能讲，但整体节奏不顺、爆点不突出、原片保留不合理。

这套系统不是“先按固定时长切开再补救”，而是：
先找到真实边界，再理解整片，再写每段，再校验，再合成。

二、最重要的总原则

原则 1：检测阶段不先把视频真实切成固定时长片段。
系统不能先把视频硬切成 2 秒一段、3 秒一段，再分别检测。
正确做法是：
先沿整条视频时间轴做检测，再根据检测结果生成真实片段。

原则 2：先切对，再看对，再写对。
先把片段边界尽量做对，再去理解内容，再去生成文案。
如果片段本身就是错的，后面理解和文案都会被污染。

原则 3：先理解整条视频的大纲，再决定每段怎么讲。
系统不能只是一段一段各自生成文案。
必须先知道整条视频在讲什么，再给每段安排讲法。

原则 4：视频内容是事实来源，文案和提示词不是事实来源。
用户提示词、风格要求、已有文案，只能影响表达方式，不能决定视频事实。

原则 5：默认只修问题段，不默认整片重跑。
如果第 8 段出问题，就优先修第 8 段。
不要因为一段出错就把整条视频全部重来。

原则 6：用户看到的所有关键处理步骤都必须在控制台打印中文日志。
每一步都要打印：
- 开始
- 过程
- 完成
- 警告
- 错误
- 回退
并且必须是中文。

三、系统完整流程

原始视频
→ 用户控制与模型配置层
→ 参考图片与命名增强层
→ 视频基础索引层
→ 完整时间轴多尺度检测层
→ 双引擎边界检测层
→ 边界融合与真实边界生成层
→ 动态片段生成层
→ 事件合并层
→ 全局视觉理解层
→ 全局剧情结构层
→ 片段映射层
→ 关键帧 / 关键片段选择层
→ 局部视觉理解层
→ 结构化摘要层
→ 文案生成层
→ 一致性校验层
→ 自修复 / 局部重跑层
→ TTS 层
→ 原片比例控制层
→ 合成层
→ 导演层全局优化
→ 全局时间轴验收层

下面按顺序解释每一层。

四、用户控制与模型配置层

这一层是所有用户可调参数和模型接入配置的入口。

这一层负责：
1. 让用户选择内容类型
2. 让用户输入提示词，影响文案风格和整片编排
3. 让用户设置原片保留比例
4. 让用户配置视觉理解模型、文案模型、TTS 模型
5. 让用户填写多个 API 地址
6. 让用户测试 API 是否可连通
7. 让用户选择本地模型或第三方模型
8. 让用户设置批处理并发数量、重跑次数、校验强度

为什么必须有这一层：
因为系统不应该把模型服务写死，也不应该把用户风格偏好写死。
这一层负责“用户想要什么风格、想怎么接模型、想怎么控制成片”。

这一层不负责什么：
这一层不负责判断视频事实。
用户提示词不能决定：
- 哪一帧属于哪个事件
- 谁一定是主体
- 哪个边界一定真实
这些仍然由后面的检测和理解层决定。

这一层可给用户开放的控制项包括：
- 内容类型
- 文案风格
- 爆点是否优先前置
- 原片保留比例
- 是否保留更多原片在高潮段
- 文案密度
- 语速
- 校验严格度
- 最大重跑次数
- 模型 API 地址
- 本地模型地址
- API 测试按钮
- 批处理并发数

技术栈：
- 前端配置界面
- 后端配置管理
- API 路由与健康检查
- 本地模型适配器
- 多地址故障切换逻辑

控制台日志要求：
- 打印“开始读取用户配置”
- 打印内容类型、风格、原片比例、模型来源
- 打印 API 测试是否成功
- 打印“用户配置加载完成”

五、参考图片与命名增强层

这一层允许用户上传图片并为图片命名，用来增强视觉理解。

用户可以上传：
- 角色图片
- 物体图片
- 场景图片
- 道具图片
并且给它们起名字。

例如：
- 哆啦A梦
- 大雄
- 大雄爸爸
- 杰瑞
- 汤姆
- 红色跑车
- 办公室场景

为什么必须有这一层：
因为很多自动理解错误，来自角色名不稳定。
系统可能把同一个角色写成：
- 男人
- 爸爸
- 上司
- 主角
参考图片和命名，可以帮助系统在后续理解中更稳定地对齐名称。

这一层的作用是：
- 增强主体识别
- 增强角色命名一致性
- 增强局部理解和全局角色状态流

这一层不负责什么：
它不是绝对真值层。
如果参考图本身有误，系统仍然要允许低置信输出，不允许盲信。

技术栈：
- 图片上传与缓存
- 参考图库管理
- 图像嵌入或参考匹配增强
- 传给视觉模型作为补充上下文

控制台日志要求：
- 打印“开始加载参考图片与命名”
- 打印参考图片数量
- 打印命名数量
- 打印“参考增强层加载完成”

六、视频基础索引层

这一层是整套系统的基础设施层。

它做的事情是：
1. 读取视频总时长
2. 读取帧率
3. 读取分辨率
4. 建立帧号和时间戳的映射
5. 建立缓存路径索引
6. 可选地缓存基础图像特征

为什么必须有这一层：
后面所有模块都要不停问这些问题：
- 3.72 秒对应哪一帧
- 第 188 帧是什么时间
- 这一段缓存在哪
- 这一段之前是否已经算过光流、直方图、关键帧

如果没有统一索引，后面每个模块都会各算各的，导致：
- 慢
- 容易出错
- 不利于局部重跑
- 不利于排查问题

这一层的本质：
不是理解内容，而是建立统一时间坐标和缓存坐标。

技术栈：
- ffprobe
- FFmpeg
- OpenCV
- 本地缓存管理

控制台日志要求：
- 打印“开始建立视频基础索引”
- 打印时长、帧率、分辨率、总帧数
- 打印缓存目录
- 打印“视频基础索引建立完成”

七、完整时间轴多尺度检测层

这是本方案里最容易被误解的一层，必须说清楚。

这一层不是先把视频切成固定片段。
这一层做的是：
沿着完整视频时间轴，连续地、分层次地找变化。

它的正确理解是：
先在整条时间轴上检测哪里可能发生了内容变化，
再决定哪些地方应该切。

它通常分两层：

第一层，粗尺度检测。
作用：
快速找到“可能有变化的区域”。

第二层，细尺度检测。
作用：
对粗尺度发现的区域做更精确定位，找到真实切点。

为什么必须有这一层：
因为如果一开始就真实切成固定 2 秒片段，那么 2 秒内部发生的变化就已经被错误打包了。
例如 2.0 到 4.0 秒这段里，真实切点可能在 3.72 秒。
如果先切成 2.0 到 4.0，后面再理解，就已经把错误混进去了。

这层的本质：
它不是在“切片”，而是在“找哪里该切”。

技术栈：
- OpenCV
- PySceneDetect
- 可选更强镜头边界检测模型
- 时间窗口扫描逻辑

控制台日志要求：
- 打印“开始完整时间轴多尺度检测”
- 打印粗尺度窗口参数
- 打印细尺度窗口参数
- 打印检测到的疑似变化区数量
- 打印“完整时间轴多尺度检测完成”

八、双引擎边界检测层

这一层负责判断：
在某个时间位置附近，真实边界到底应不应该切、切在哪里。

它分两套引擎。

第一套：主体切换引擎

这一套专门解决：
下一段主体提前出现在当前段尾部。

它检测的不是“画面整体变没变”，而是：
- 核心主体变没变
- 主体集合变没变
- 新主体是不是稳定进入画面
- 旧主体是不是退出核心区域
- 当前画面中心讲的还是不是同一个主体

为什么必须有它：
因为很多时候镜头没完全切，场景也没明显变化，但叙事主体已经换了。
这正是音画错位的核心来源之一。

技术栈：
- YOLO11
- BoT-SORT
- ByteTrack（可替代）
- 可选参考图增强
- 高难片段可按需触发更强分割/跟踪

第二套：镜头 / 场景切变引擎

这一套负责判断：
- 是否硬切镜头
- 是否软转场
- 是否整体画面变化明显
- 是否运动模式突变
- 是否场景已经切换

为什么必须有它：
因为不是所有边界都来自主体切换。
有些时候主体还是同一个，但镜头或场景已经明显切到新的叙事阶段。

技术栈：
- PySceneDetect
- OpenCV 直方图比较
- OpenCV 光流
- OpenCV 帧差
- 可选更强镜头切分模型

这两套引擎必须共同工作，不能只用一套。

控制台日志要求：
- 打印“开始主体切换引擎”
- 打印主体候选点数量
- 打印“主体切换引擎完成”
- 打印“开始镜头/场景切变引擎”
- 打印场景候选点数量
- 打印“镜头/场景切变引擎完成”

九、边界融合与真实边界生成层

前一层可能会给出很多候选点。
这一层负责把它们融合成“真正可用的边界”。

它要做的事包括：
1. 合并时间很接近的边界候选
2. 过滤明显无效点
3. 给每个边界打置信度
4. 标记边界来自什么触发类型
5. 生成真正的真实边界列表

为什么必须有这一层：
因为主体引擎和镜头引擎可能给出不同的答案。
必须有一层统一裁决，不然系统会混乱。

这一层必须遵守的规则：
1. 主体提前入画问题优先于自然镜头边界问题
2. 单一检测器不能直接裁决
3. 低置信边界不能强切
4. 无有效点可以保留当前边界，不要为了切而切

技术栈：
- 规则融合器
- 时间邻域合并逻辑
- 置信度评分逻辑

控制台日志要求：
- 打印“开始边界融合”
- 打印每类候选点数量
- 打印最终真实边界数量
- 打印低置信边界数量
- 打印“边界融合完成”

十、动态片段生成层

这一层负责把“真实边界”变成“真正可用的非均匀片段”。

它做的事情很简单但很关键：
根据真实边界，生成一段一段真正的动态片段。

例如不再是：
- 0 到 2
- 2 到 4
- 4 到 6

而可能变成：
- 0 到 1.42
- 1.42 到 2.68
- 2.68 到 3.50
- 3.50 到 6.00

为什么必须有这一层：
因为真实视频内容不是固定长度生成的。
没有这一步，系统就仍然停留在假边界层面。

这一层必须保证：
- 无重叠
- 无空缺
- 首尾连续
- 每段都有合法开始和结束

技术栈：
- 时间轴生成逻辑
- 片段合法性检查器

控制台日志要求：
- 打印“开始生成动态片段”
- 打印片段数量
- 打印最短片段和最长片段
- 打印“动态片段生成完成”

十一、核心区 / 容错区修正层

这一层是“边界安全护栏”。

它解决的问题是：
如果边界需要修，到底优先修哪儿，怎么修，修多少，才不容易破坏当前段内容。

它会给每个片段定义两个区：

核心区：
这一段里主要内容所在的区，尽量不动。

容错区：
这一段更容易出问题、尤其容易混入下一段内容的区，优先在这里修边界。

为什么必须有这一层：
因为大部分错位，不是整段都错，而是尾巴混入下一段。
如果没有这层，系统可能会整段乱动，反而破坏原本正确的部分。

必须明确：
这里不能写死成“前 80% 不动，后 20% 才可动”。
正确做法是动态计算。
计算因素包括：
- 片段总时长
- 内容类型
- 风险等级
- 配音有效时长占比
- 是否动作段
- 是否多主体段

这一层还要处理配音红线：
如果画面应该在 3.72 秒切，但配音到 3.80 秒才讲完，不能简单粗暴一律拖画面。
正确顺序是：
1. 先尝试压缩文案
2. 再尝试小幅调节语速
3. 只有差值极小时，才允许轻微延后画面
4. 不允许为了等配音过度拖画面，造成新的画面语义错位

技术栈：
- 时间规则引擎
- 动态区间计算逻辑
- 文案时长估计器
- 语速微调控制逻辑

控制台日志要求：
- 打印“开始核心区 / 容错区修正”
- 打印每段核心区和容错区
- 打印因配音红线触发的修正次数
- 打印“核心区 / 容错区修正完成”

十二、事件合并层

这一层负责把镜头级片段提升成事件级片段。

它要做的是：
把一堆属于同一个语义事件的碎镜头合并成更完整的事件段。

为什么必须有这一层：
因为镜头变化不等于事件变化。

例子：
动作番里一个完整攻击过程，可能切了 4 个镜头。
如果不合并，文案会变得非常碎。
电影里一个完整的情绪反应链，也可能被反打镜头切碎。
如果不合并，文案会失去完整性。

这一层必须支持的内容类型规则：
- 对话番：减少反打碎裂
- 动作番：合并动作链
- 无对白动画：合并动作因果链
- 电影 / 电视剧：保留情绪推进完整性

技术栈：
- 规则引擎
- 主体变化信息
- 运动变化信息
- 场景变化信息
- 内容类型策略

控制台日志要求：
- 打印“开始事件合并”
- 打印合并前镜头段数量
- 打印合并后事件段数量
- 打印“事件合并完成”

十三、全局视觉理解层

这一层必须讲清楚，不能有歧义。

全局视觉理解不是：
- 把整条视频上万帧逐帧全看一遍
- 对全片每一帧做高精度理解

全局视觉理解的正确含义是：
在完成动态片段生成和事件合并之后，
基于每个事件段的代表性视觉信息，
先对整条视频建立一个全片级剧情理解结果。

它看的不是所有帧，而是每个事件段的代表信息，例如：
- 起点代表帧
- 峰值代表帧
- 结果代表帧
- 必要时的反应帧
- 极少数高风险段的关键短片段

为什么必须有这一层：
因为如果没有全局理解，系统只能逐段理解，最后容易变成：
- 单段都对
- 整片像流水账
- 角色称呼漂移
- 爆点和铺垫顺序不顺

这一层输出的是：
- 总剧情摘要
- 全局事件树
- 角色状态流
- 冲突推进结构
- 高价值片段列表

技术栈：
- VLM API
- 事件段代表输入
- 上下文聚合逻辑

控制台日志要求：
- 打印“开始全局视觉理解”
- 打印事件段数量
- 打印代表帧数量
- 打印主剧情摘要
- 打印“全局视觉理解完成”

十四、总剧情 / 全局事件树 / 角色状态流

这一层是全局视觉理解的产物层。

它必须输出以下内容：

1. 总剧情摘要
作用：
概括整条视频主要在讲什么。

2. 全局事件树
作用：
定义主要事件之间的前后关系、因果关系、主次关系。

3. 角色状态流
作用：
说明角色在不同事件中的状态变化，避免后面文案称呼乱跳。

4. 高价值片段列表
作用：
找出哪些段是爆点、反转、情绪高潮、搞笑结果。

为什么要把这层单独列出来：
因为后面的文案生成、导演层、原片优先、压缩跳段都依赖它。

控制台日志要求：
- 打印“开始生成总剧情 / 事件树 / 角色状态流”
- 打印主角色数量
- 打印事件树节点数量
- 打印高价值片段数量
- 打印“总剧情结构生成完成”

十五、局部片段映射到全局剧情

这一层负责给每个片段确定它在整片中的作用。

它要回答的问题是：
- 这段是铺垫吗
- 这段是推进吗
- 这段是高潮吗
- 这段是反转吗
- 这段是收尾吗

为什么必须有这一层：
同样一段内容，在整片不同位置讲法是不一样的。
没有这层，文案就容易“每段都像重新开头”。

技术栈：
- 规则映射
- 事件树读取
- 角色状态流读取

控制台日志要求：
- 打印“开始片段映射到全局剧情”
- 打印每段角色标签统计
- 打印“片段映射完成”

十六、关键帧 / 关键片段选择层

这一层负责从每个事件段选最有价值的输入，给局部视觉理解使用。

它不是平均抽很多帧。
它要尽量选：
- 起点帧
- 峰值帧
- 结果帧
- 必要时反应帧

高风险段可升级成关键短片段。

为什么必须有这一层：
如果抽太少，看不懂；
抽太多，成本高，还会有大量重复信息。

技术栈：
- OpenCV
- FFmpeg 抽帧
- 帧差与光流辅助
- 去重逻辑

控制台日志要求：
- 打印“开始选择关键帧 / 关键片段”
- 打印每段抽取数量
- 打印高风险段数量
- 打印“关键帧 / 关键片段选择完成”

十七、局部视觉理解层

这一层是在全局理解已经有了之后，对具体片段做更细的补充理解。

它负责回答：
- 当前片段主要是谁
- 当前片段主要发生了什么
- 谁在反应
- 当前情绪是什么
- 当前结果是什么

为什么必须有这一层：
全局层只给整片大纲，不足以直接写好每段文案。
局部层负责补足细节。

技术栈：
- VLM API
- 关键帧 / 关键短片段
- 局部上下文

控制台日志要求：
- 打印“开始局部视觉理解”
- 打印当前片段编号
- 打印主体、动作、结果摘要
- 打印置信度
- 打印“局部视觉理解完成”

十八、结构化摘要层

这一层负责把局部视觉理解结果转换成标准化中间表示。

必须输出的核心字段包括：
- 主体
- 动作
- 对手 / 对象
- 场景
- 情绪
- 主事件
- 结果
- 置信度
- 不确定说明

为什么必须有这一层：
因为视觉理解和文案生成必须分离。
如果不分离，后续很难重写文案、很难局部重跑、很难做统一校验。

这一层的意义是：
- 把“看懂”标准化
- 让“说出来”单独交给文案层

控制台日志要求：
- 打印“开始生成结构化摘要”
- 打印摘要关键字段
- 打印置信度
- 打印“结构化摘要生成完成”

十九、文案生成层

这一层负责真正写文案。

输入是：
- 全局剧情
- 片段在整片中的角色
- 局部结构化摘要
- 用户风格提示词
- 时长限制

输出是：
- 当前片段文案
- 片段角色标签
- 语速建议
- 是否原片优先

为什么必须放到这个阶段：
因为文案是表达层。
它必须建立在“已经看懂视频”的基础上，而不是反过来指导事实判断。

这一层要完成的工作：
1. 文案长度服从片段时长
2. 高价值段允许少讲、先放原片
3. 低价值重复段允许压缩或跳讲
4. 文案风格受用户提示词影响
5. 文案必须服从全局剧情主线

技术栈：
- 文案大模型
- 风格模板系统
- 片段角色模板系统
- 用户提示词注入层

控制台日志要求：
- 打印“开始生成文案”
- 打印当前片段编号
- 打印文案长度
- 打印片段角色
- 打印“文案生成完成”

二十、一致性校验层

这一层负责检查：
当前片段讲得对不对、贴得对不对、时长对不对。

必须至少校验三项：
1. 主体合规性
2. 配音时长合规性
3. 文案-画面语义一致性

为什么必须有这一层：
因为上游不可能完全零误差。
这一层是防止明显错误直接进入合成。

这一层必须输出：
- 是否通过
- 失败原因码
- 诊断信息

技术栈：
- 规则校验器
- 图文相似度打分
- 高风险段廉价 VLM 复核

控制台日志要求：
- 打印“开始一致性校验”
- 打印当前片段编号
- 打印校验结果
- 打印失败原因码
- 打印“一致性校验完成”

二十一、自修复 / 局部重跑层

这一层负责修问题段，而不是重跑整片。

固定修复顺序：
1. 重写文案
2. 补关键帧重跑局部视觉理解
3. 合并相邻段
4. 回到边界层重切

为什么这个顺序最合理：
因为从便宜到贵、从小改到大改。

它必须限制：
- 单段最大重跑次数
- 死循环重跑
- 无休止修复

达上限后允许进入保守输出模式。

控制台日志要求：
- 打印“开始自修复 / 局部重跑”
- 打印当前片段编号
- 打印重跑类型
- 打印重跑次数
- 打印“自修复 / 局部重跑完成”

二十二、TTS 层

这一层负责把最终文案变成音频。

这一层必须遵守一个关键原则：
TTS 不是拿来拯救文案过长问题的主要手段。
主要手段仍然是：
- 文案控制
- 文案重写
- 语义压缩

TTS 只做：
- 生成语音
- 小幅调速
- 输出分段音频

控制台日志要求：
- 打印“开始生成配音”
- 打印当前片段编号
- 打印文案长度
- 打印音频时长
- 打印“配音生成完成”

二十三、原片比例控制层

这一层负责用户对原片播放比例的控制。

用户可以控制：
- 整体原片保留比例
- 高价值段原片保留更多
- 爆点段先放原片多久
- 低价值段减少原片保留

为什么必须有这一层：
因为用户明确要控制原片播放比例。
这个需求不能只靠导演层隐式完成，必须单独有策略层。

这一层只影响：
- 导演层
- 合成层

不影响：
- 边界检测
- 事实判断
- 事件合并

控制台日志要求：
- 打印“开始原片比例控制计算”
- 打印整体原片比例
- 打印高价值段原片保留情况
- 打印“原片比例控制计算完成”

二十四、合成层

这一层负责把：
- 视频片段
- 配音
- 字幕
- 原片保留策略

最终拼成视频。

原则：
- 默认普通硬切
- 不默认复杂转场
- 原片优先段允许先放原片再接解说
- 至少做到段级严格对齐

技术栈：
- FFmpeg

控制台日志要求：
- 打印“开始视频合成”
- 打印总片段数
- 打印字幕状态
- 打印“视频合成完成”

二十五、导演层全局优化

这一层负责把整条视频从“技术正确”优化到“内容更像人剪”。

它负责：
1. 内容价值判断
2. 原片优先判断
3. 自动压缩与跳段
4. 全局叙事编排
5. 高价值片段强化处理
6. 用户提示词对片段排序和强调点的影响

为什么必须有这一层：
因为技术层只解决“对不对”，导演层解决“好不好看”。

这一层要回答的问题包括：
- 哪些段该重点讲
- 哪些段可以压缩
- 哪些段应该跳过
- 哪些段应该先放原片
- 爆点是不是要提前
- 节奏是不是要更快

控制台日志要求：
- 打印“开始导演层优化”
- 打印高价值段数量
- 打印压缩段数量
- 打印跳过段数量
- 打印“导演层优化完成”

二十六、全局时间轴验收层

这是最后一道总验收。

它负责检查整条视频是否真的可以交付。

必须检查：
1. 有没有重叠
2. 有没有空缺
3. 顺序是否错乱
4. 是否存在无效极短段
5. 整体节奏是否合理
6. 开头、推进、高潮、收尾是否完整

为什么必须有这一层：
因为前面每一层都可能局部正确，但最后只有这一层能检查整片是否自洽。

控制台日志要求：
- 打印“开始全局时间轴验收”
- 打印重叠数量、空缺数量、无效段数量
- 打印验收是否通过
- 打印“全局时间轴验收完成”

二十七、批量并发任务调度层

这一层负责批量处理：
- 多段视觉理解
- 多段文案生成
- 多段 TTS

为什么必须有：
因为你要支持批量 API 处理，提高吞吐。

它必须支持：
- 批量提交
- 并发控制
- 失败片段单独重跑
- 成功片段直接保留

这一层只改变调度方式，不改变单段逻辑。

控制台日志要求：
- 打印“开始批量任务调度”
- 打印批次数量
- 打印并发数量
- 打印失败片段数量
- 打印“批量任务调度完成”

二十八、AI 开发必须遵守的规则

1. 不允许先把视频真实切成固定 2 秒片段再检测
2. 必须沿完整时间轴做检测
3. 边界修正必须先于局部视觉理解
4. 全局视觉理解必须先于局部文案生成
5. 事件合并必须独立存在
6. 导演层必须独立存在
7. 每个模块都必须定义目标、输入、输出、规则、异常、回退
8. 默认只重跑问题段
9. 所有低置信结果必须显式输出
10. 所有关键步骤必须打印中文控制台日志
11. 不允许静默失败
12. 不允许把用户提示词当成视频事实来源
13. 不允许把参考图片当成绝对真值
14. 不允许默认整片重跑
15. 不允许省略全局时间轴验收层
16.UI框架必须使用shadecnUI，风格必须是简约风，高大上，现代化的，参照fronted-skil1,蔡止胡乱写前端UI
17.必须要前后端分离，不允许出现脏代码，不允许一个代码文件出现上千行代码。
18.保证高内聚低耦合准则
19.该软件只是个人使用，无需考虑高并发等商业需求

二十九、最后一句总定义

这套方案的正确理解是：

先在完整时间轴上检测真实变化，再生成动态片段；
再把镜头段合并成事件段；
再先理解整条视频的大纲和角色关系；
再根据全局剧情给每个片段生成局部文案；
再通过校验、自修复、TTS、原片比例控制、导演层优化和全局验收，输出最终成片。

这是一套从“视频事实”出发，再到“内容表达”和“成片控制”的完整闭环系统。
